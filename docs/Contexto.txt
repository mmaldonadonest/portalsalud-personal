## 3. Base de datos

### Tablas en MariaDB (fuente)

#### `files`
```sql
id            INT(10) PK AUTO_INCREMENT
nss           VARCHAR(50)
name          VARCHAR(50)    -- ⚠️ encoding corrupto ~0.4% filas (latin1/utf8)
date_upload   DATETIME
url           LONGTEXT       -- ‼️ NO es URL: almacena el PDF completo en base64
type          VARCHAR(50)    -- valores: examen_medico|nota_medica|laboratorio|
                             --          nota_incapacidad|''|[hash_MD5_32chars]
```

**Estadísticas reales (muestra 500 filas, 161 NSS):**
- `url` en base64: promedio 539 KB, mediana 514 KB, p90 932 KB, máx ~2 MB por fila
- 11 filas con `url` vacío (huérfanas)
- `type` distribución: examen_medico 68%, nota_medica 11%, laboratorio 10%, nota_incapacidad <1%, vacío <1%, hashes MD5 9.4% (origen a confirmar en PHP)
- **17 GB del backup completo** es consistente: binarios base64 × miles de registros

**Decisión de diseño recomendada:** migrar `url` a `BLOB` en Oracle, decodificando base64 en el ETL → ahorra ~33% de espacio vs mantener CLOB+base64.

#### `tags`
```sql
id       INT(255) PK AUTO_INCREMENT
nss      VARCHAR(255)
type     VARCHAR(50)    -- nombre del campo (Drenaje, CCA, nombre1, giro1...)
content  LONGTEXT       -- valor del campo
```

**Estructura EAV — 47 campos fijos por NSS:**
```
Clínicos (7):   Drenaje, CCA, VOZCLARAYFUERTE, STPO2,
                edad_inicio_laborar, cantidad_trabajos, pension

Historial laboral x4 trabajos (40):
                nombre1..4, giro1..4, puesto1..4, turno1..4,
                antiguedad1..4, salida1..4, descripcion1..4,
                riesgos1..4, epp1..4, observaciones1..4
```

**⚠️ BUG CRÍTICO detectado en PHP:**
El código hace INSERT sin verificar existencia (nunca hace UPDATE), generando bloques duplicados idénticos. Un solo NSS puede tener 7 copias exactas del mismo bloque de 47 filas. La tabla en producción tiene registros masivamente inflados.

**Regla ETL para deduplicar:**
```sql
-- Solo conservar el bloque con ID más alto por NSS
SELECT * FROM tags t1
WHERE t1.id IN (
  SELECT MAX(t2.id) FROM tags t2
  WHERE t2.nss = t1.nss AND t2.type = t1.type
)
```

**⚠️ Encoding corrupto en `content`:** campos `antiguedad1` y `observaciones1` tienen tildes corruptas (`aÃ±os` → `años`, `presiÃ³n` → `presión`). Corrección ETL:
```python
content.encode('latin1').decode('utf-8')
```

**⚠️ Caso edge:** un NSS con 46 campos (falta `Drenaje`) — el ETL debe manejar campos faltantes con defaults.

## 6. Riesgos críticos identificados

1. **API externa no documentada** en red local para NSS/autenticación — **Step 0 bloqueante** antes de iniciar desarrollo Java.
2. **SQL injection** en PHP legacy por concatenación de strings.
3. **Bug activo en `pretest.php`** (comportamiento anómalo documentado).
4. **Archivo JS dashboard vacío** — lógica de estado frontend a descubrir.
5. **Latencias 6–7s** en endpoints PDF (`viewTypeExped`, `expedienteFilePrint`, `expeFileNotaMedPdf`, `expFileNotaIncp`) — necesitan tuning específico en Java+Oracle.
6. **BUG en tabla `tags`**: INSERT sin UPDATE → duplicación masiva. ETL debe deduplicar.
7. **Encoding mixto** en `files.name` y `tags.content` (latin1/utf8).
8. **`files.url` es base64 de PDF binario**, no una ruta — implica decisión de diseño BLOB vs CLOB en Oracle.
